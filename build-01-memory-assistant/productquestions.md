Memory boundary transparency — Does your user know when the system has forgotten something they said earlier, and if so, how does the product communicate that without breaking their trust?
We embed a context meter in the chat UI that shows turn count in plain language, shifts from green to amber to red as the window fills, and surfaces a pin prompt on individual messages before they fall out of the window, so the user always knows what the system is holding and has a clear action to take before anything is lost.

Which memory strategy — sliding window, summarization, or token budget — maps to which user tier, and what is the unit economics justification for each?
Sliding window depends on window length, and say if it is set at 4 turn, N tokens per call, that is X cents per 100 conversations. This might be a good strategy for a free tier. 
Summarization adds a periodic model call; amortised across turns, not per message, making it more expensive than window but predictably so. Right for paid tier users who need longer memory and are willing to pay for it.
Token budget, where you set the amount of tokens, is probably a good strategy for enterprise customers that need control and manage costs. They can put that in there contract
